m3u8 解释

1.M3U8是一种可扩展的播放列表文件格式。它是一个包含UTF-8编码文字的m3u播放列表。m3u是包含媒体文件URL的一个事实上的播放列表标准。这种格式被用来作为HTTP Live 媒体流索引文件的格式。

2.m3u8是一种视频列表格式，里面有真正的视频的链接，另外在m3u8里面还可以再嵌套一层m3u8

3.m3u8是视频列表，编码还是h264等。

4.m3u8并非苹果独占，m3u8这种列表其实编码格式是公开的。

5.HLS 苹果推出的一种视频直播解决方案，将视频分成5-10秒的视频小段，然后用m3u8索引进行管理。使用案例：qq和微信朋友圈的URL分享。

责任链模式

  • 概念

  • 责任链模式是一种对象的行为模式。在责任链模式里，很多对象由每一个对象对其下家的引用而连接起来形成一条链。请求在这个链上传递，直到链上的某一个对象决定处理此请求。发出这个请求的客户端并不知道链上的哪一个对象最终处理这个请求，这使得系统可以在不影响客户端的情况下动态地重新组织和分配责任。

  • 代码实现

  • 处理请求的抽象类

    public abstract class Handler {
    public Handler nextHandler;
    public  void handleRequest(AbstractRequest abstractRequest)
    {
        if(getHandleLevel()==abstractRequest.getRequestLevel())
        {
            handle(abstractRequest);
        }else {
            if(nextHandler!=null)
            {
                nextHandler.handleRequest(abstractRequest);
            }else {
                System.out.println("---->  所有的处理对象都不能处理它");
            }

        }
    }
    /**
     * 每个处理者的对象的具体处理方式
     * @param abstractRequest
     */
    public abstract void handle(AbstractRequest abstractRequest);
    /**
     * 返回每个对象能处理请求的级别
     * @return
     */
    public abstract int getHandleLevel();
}

  • 处理请求实现类1

        public class Handler1 extends Handler{

    @Override
    public void handle(AbstractRequest abstractRequest) {
        System.out.println("----handle1  处理请求: "+abstractRequest.getRequestLevel());
    }

    @Override
    public int getHandleLevel() {
        return 1;
    }

}

  • 处理请求实现类2

    public class Handler2 extends Handler {

    @Override
    public void handle(AbstractRequest abstractRequest) {
        System.out.println("----handle2  处理请求: "+abstractRequest.getRequestLevel());
    }

    @Override
    public int getHandleLevel() {
        return 2;
    }

}

  • 处理请求实现类3

    public class Handler3 extends Handler {

            @Override
            public void handle(AbstractRequest abstractRequest) {
                System.out.println("----handle3  处理请求: "+abstractRequest.getRequestLevel());
            }

            @Override
            public int getHandleLevel() {
                return 3;
            }

        }

  • 请求类

    public class Request  extends AbstractRequest{

    public Request(Object object) {
        super(object);
    }

    @Override
    public int getRequestLevel() {
        return 3;
    }

}

  • 调用

    public class Client {
    public static void main(String[] args) {
        //确定链式关系
        Handler handler1=new Handler1();
        Handler handler2=new Handler2();
        Handler handler3=new Handler3();
        handler1.nextHandler=handler2;
        handler2.nextHandler=handler3;

        AbstractRequest request=new Request("请求");

        handler1.handleRequest(request);
    }
}

执行结果：—-handle3 处理请求: 3

建造者模式

###标准构建者模式

  • 定义

将一个复杂的构建与他的表示分离，使得同样的构建过程可以创建不同的表示。

  • 角色

    • 抽象建造者

    • 具体建造者

    • 设计者

    • 产品

    eg：比如要建一个房子，房子就是产品，设计者就是设计房子的，具体建造者就是工人，而工人按照先打地基，后垒墙，最后封顶的顺序，这个顺序就是抽象建造者。

###建造者模式的改进型

  • 产品

    public class LeequerRoom {
    private String root;
    private String top;
    private String wall;
    private String window;
    public void applyRoom(RoomParam param) {
        root = param.root;
        top = param.top;
        wall = param.wall;
        window = param.window;
    }

    @Override
    public String toString() {
        return "leequer 房子:" + root + ":" + wall + ":" + top+":"+window;
    }
}

    我们为了成员变量的安全，不能随意改动，删除了get，set方法，但这样谁也无法改变它了，RoomParam 是建造者中的内部类，就是用来解决成员变量无法访问的问题。

  • 具体建造者

  • 建造者模式的优点

无须关心产品对象的具体组装过程，只需指定具体建造者的类型即可。

建造者模式与抽象工厂模式有点相似，但是建造者模式返回一个完整的复杂产品，而抽象工厂模式返回一系列相关的产品；在抽象工厂模式中，客户端通过选择具体工厂来生成所需对象，而在建造者模式中，侧重于一步步构造一个复杂对象，然后将结果返回。

如果将抽象工厂模式看成一个汽车配件生产厂，生成不同类型的汽车配件，那么建造者模式就是一个汽车组装厂，通过对配件进行组装返回一辆完整的汽车。

面向切面编程AOP

AOP是Aspect-Oriented Progreming的缩写，在OOP设计中有个单一职责原则，在很多时候都不会有问题，但是当很多模块都需要同一个功能的时候，这个时候还用OOP就会很麻烦。那么AOP在Android中的应用就应运而生。使用的工具是aspectJ。

###需求

  • 计算方法执行的时间

    public void one(View view) {

    long beagin=System.currentTimeMillis();

{
    SystemClock.sleep(3000);

    Log.i(TAG,"点击 第一个按钮 执行");

}

Log.i(TAG,"消耗时间：  "+(System.currentTimeMillis()-beagin)+"ms");
}

    如果所有的方法都需要计算消耗时间，那每个方法都需要写同样的代码，一旦需求改动，所有的计算耗时的代码全部需要改动。写到这里，我在想，把计算耗时的封装在一个方法里面，做成工具类不就行了么？于是有了这样的代码：

    public void one(View view) {

    }
    duration(begin);

}
public String duration(long begin){
    Log.i(TAG,"消耗时间：  "+(System.currentTimeMillis()-begin)+"ms");
}

    那这时，需求改了，要判断是那个类输出的日志。那就是要加个类的参数，那我就需要修改这儿duration的方法，添加一个参数，那么所有调用这个方法的地方都需要修改。而且你发现，按个begin的计算，只是为了计算耗时，每个方法里面都要存在。

@BehaviorTrace("one")
public void one(View view) {

在one这个方法上方添加注解@BehaviorTrace(“one”)，首先BehaviorTrace这个不是固定的写法，这个是你创建的切点的名字，这里的名字要和切点类名相同，参数one，是对应BehaviorTrace中的 String value();方法，这是固定写法，来看看BehaviorTrace 的代码：

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.CLASS)
public @interface BehaviorTrace {

    String value();

}

a:固定写法，通过这个注解表示这个类是切面。

b：

Pointcut注解：

就是切点，即触发该类的条件。

使用了execution，

也就是以方法执行时为切点，触发Aspect类。而execution里面的字符串是触发条件，也是具体的切点。

-

"execution(@com.example.liqing.aopdemo.BehaviorTrace  * *(..))"

这个条件是所有加了BehaviorTrace注解的方法或属性都会是切点，范围比较广。也就是说，执行每一个被BehaviorTrace注解的方法，都会执行此切点的功能。

“**”表示是任意包名

“..”表示任意类型任意多个参数

C：从这里开始执行的就是被BehaviorTrace注解的方法的内容，前面的都是先执行的 到e 结束。后面的代码就是执行完被注解的方法后执行的。

d：这些被注释掉的代码在问题出在

behaviorTrace=_method.getAnnotation(BehaviorTrace.class);

该getAnnotation()方法在Android 5.0以上是能获取到注解实例的，但是在Android 5.0以下则会返回null。
