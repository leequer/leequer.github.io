产品经理课程

摘要

1.平庸的产品经理：排期，推进，逼着开发尽快实现！
2.卓越的产品经理：用户的目标是什么？为什么要提这个需求？是否做？如何实现？

#能力1.分清用户需求和用户目标

用户说：我要去外地，想要一个更快的马。于是你找到一匹好马

我们说这个并不没有很好的分析用户的目标，用户的目标是更快，而不是马。更快的马是用户认知范围内的一种
解决方案，他的目标是更快。而不是一匹马。当他看到有汽车跑的比马更快的时候，他就会抱怨这是什么产品。产品经理需要做典型用户需求分析，找出真正用户的痛点。所有用户的需求都是原始需求，只有当产品经理分析出其中的用户目标才是我们需要解决的问题。飞机，高铁，更快的交通工具发明出来，解决的一个问题就是更快。

分清用户需求和目标的简单区别，“科技再怎么变，人性没有变”，目标不会随着时间改变。

##常见的产品经理的错误：

把细小的需求强加到用户身上，有的产品经理认为粉红色会讨女客户喜欢，而并不是所有女客户都喜欢，这个粉红色就是强加到客户身上的需求。

用户都是傻瓜，他们不知道自己要什么，no，用户不是专业人士，他们不知道所有的解决方案，但是每个用户都知道自己要什么样的产品，否则他怎么会选你的产品。

人性的需求是否被满足，通俗的说法就是“抓住了痛点”，这就是要求产品经理抛开现象看本质，通过研究典型用户，模拟典型场景，抛开现有的技术手段，寻找人性中最本质的需求，并通过产品设计，满足用户的人性需求

##马斯洛的人性的需求，从低到高，生理（吃喝玩乐，大众点评，百度外卖），安全，社交（不孤独，不寂寞，微信，qq），尊重，自我实现，越是高级的需求，越不会被用户抛弃，微信在满足大家社交的需求的基础上，向下满足支付，购买，满足人们的生理需求，这样更牢固。你让百度外卖做社交，肯定做不起来。但是让微信做支付，很快有人来用。一个道理。点赞和分享就是往上满足人们的更高需求。

能力要求 2 ：重塑和分解需求的能力

##产品经理在讲解自己的设计和计划之前，要把这个设计的初衷，就是自己捕获到的客户目标说一下，和增加这项功能的原因和考虑。以得到大家的认同感。否则我们不知道你为什么这么设计，出于什么原因设计。开发听了也一头雾水。

#能力要求 3 ：收集资料和整理数据的能力

用数据说话：避免争吵，增加方案说服力。

1）掌握硬件、设备、系统、软件新技术的进步以及动向

2）掌握市场发展和占比等数据的变化

3）看准产业链的发展趋势和新机会

4）研究用户喜好和市场热点变化

5）研究新的交互方式、用户体验设计方式、设计流行趋势

CMake语法1-基本使用

#1.简介

####CMake是一个跨平台的安装（编译）工具，可以用简单的语句来描述所有平台的安装(编译过程)。他能够输出各种各样的makefile或者project文件，能测试编译器所支持的C++特性,类似UNIX下的automake。只是 CMake 的组态档取名为 CMakeLists.txt。Cmake 并不直接建构出最终的软件，而是产生标准的建构档（如 Unix 的 Makefile 或 Windows Visual C++ 的 projects/workspaces），然后再依一般的建构方式使用。这使得熟悉某个集成开发环境（IDE）的开发者可以用标准的方式建构他的软件，这种可以使用各平台的原生建构系统的能力是 CMake 和 SCons 等其他类似系统的区别之处。

#2.使用

##2.1 创建必要的文件夹

###@build文件夹：编译生成的内容在此文件夹下。

###@src文件夹：放源文件

###@include文件夹：放头文件

##2.2 新建CMakeLists.txt (必须是这个名字)
写入内容：

#支持最小的编译版本号
cmake_minimum_required(VERSION 2.6)
#目标：生成myapp
PROJECT(myapp)
#导入头文件所在文件夹
INCLUDE_DIRECTORIES(
include
)
#指定源文件的目录
AUX_SOURCE_DIRECTORY(src DIR_SRCS)
#把可执行的文件放大哪个目录下  SET  方法设置变量
SET(TEST_MATH ${DIR_SRCS})
ADD_EXECUTABLE(${PROJECT_NAME} ${TEST_MATH})

#3.生成Makefile文件

进入到build文件夹下 执行cmake .. 生成

看来cmake作用就是生成Makefile文件

#4.生成可执行文件

在build文件夹下执行make命令 生成myapp 然后执行 ./myapp

Makefile语法2-自定义方法，递归式展开

#1自定义方法

#自定义方法myfun 两个参数 
myfun=$2 $1
#自定义方法myfun_ret 调用myfun方法 传入第二个参数20 第一个参数10 注意顺序 是反着的
myfun_ret=$(call myfun,20,10)
#输出
test:
    @echo $(myfun_ret)

###$>执行make test 结果是10 20

#2递归式展开 是用等号 =

str2是str1赋值的，递归展开寻找

str2=$(str1)
str1=hello
test:
    @echo $(str2)

###$>执行make 或者 make test 结果是hello

str2=$(str1)
str1=hello
str1=world
test:
    @echo $(str2)

###$>这个会输出什么呢？ 结果是world

#3直接式展开 是分号等号 ：=

str1:=android
str1:=ios
str2:=$（str1）
test:
    @echo $(str2)

###$>输出结果 ios

Makefile 语法 一

#1简介

Linux 环境下的程序员如果不会使用 GNU make 来构建和管理自己的工程，应该
不能算是一个合格的专业程序员，至少不能称得上是 Unix 程序员。在 Linux（unix）环
境下使用 GNU 的 make 工具能够比较容易的构建一个属于你自己的工程，整个工程的
编译只需要一个命令就可以完成编译、连接以至于最后的执行。不过这需要我们投入一
些时间去完成一个或者多个称之为 Makefile 文件的编写。此文件正是 make 正常工作
的基础。Makefile只能在Linux下使用。

####编译整个工程你所要做的唯一的一件事就是在 shell 提示符下输入 make 命令。

####make 在执行时，需要一个命名为 Makefile 的文件。也就是说你要有个文件叫Makefile 其它什么都不行。

下面是一个Makefile文件的内容：

Makefile

#mainapp 是 目标文件 编译这个mainapp文件 需要有后面的文件
mainapp:main.o plus.o minus.o multi.o divi.o
    gcc main.o plus.o minus.o multi.o divi.o -o mainapp
#而要得到main.o这个文件需要main.c 这个文件
mian.o:main.c
#而main.c 要执行一个gcc的编译命令才能生成main.o  那么一个tab键就是执行命令的意思。可以执行shell命令 一个命令一行
    gcc -c main.c
plus.o:plus.c
    gcc -c plus.c
minus.o:minus.c
    gcc -c minus.c
multi.o:multi.c
    gcc -c multi.c
divi.o:divi.c
    gcc -c divi.c
clean:
    rm -f *.o

###$>执行make命令运行结果 生成各个的.o 文件和 mainapp这个可执行文件

###$>执行./mainapp 运算结果

###$>执行make clean 清楚生成的.o文件

###$>如果已经生成了mainapp文件，再执行make 是无法再次生成的 要对某个文件修改才能再次编译，比如执行touch main.c 在执行make就可以再次编译。

#2.优化

这样写起来并不简单啊 ，是否可以优化？

在实际工作中大家都比较认同的方法是，使用一个变量 “objects”、“OBJECTS”、“objs”、“OBJS”、“obj”或者“OBJ”来作为所有的.o 文 件的列表的替代。在使用到这些文件列表的地方，使用此变量来代替

上面的文件内容可改为：

#object 是变量
objects=main.o plus.o minus.o multi.o divi.o
mainapp:$(objects)
    gcc $(objects) -o mainapp
#%.o 所有的输出文件  %.c所有的依赖文件
%.o:%.c
#$^ 输出文件 $@依赖文件
    gcc -c $^ -o $@
clean:
    rm -f *.o

#3.再优化

#一般我们可以使用“$(wildcard *.c)”来获取工作目录下的所有的.c 文件列表
source=$(wildcard *.c)
#；可以使用“$(patsubst %.c,%.o,$(wildcard *.c))”，首先使用“wildcard” 函数获取工作目录下的.c 文件列表；之后将列表中所有文件名的后缀.c 替换为.o。这样 我们就可以得到在当前目录可生成的.o 文件列表
objects=$(patsubst %.c,%.o,$(source))
mainapp:$(objects)
    gcc $^ -o $@
%.o:%.c
    gcc -c $^ -o $@
clean:
    rm -f *.o

密码概要

本标签中的文章记录阅读《图解密码技术》的笔记。

1密码的应用

  • a发送邮件给b，在这个过程中，a就是sender，b就是receiver。这个过程中有可能有eve监听或者截获了邮件。这个eve不一定是人，也可以是一段监听的程序。
  • 于是，a发送邮件的时候进行了加密，加密前的叫原文，加密后的叫密文，b收到后要解密才能看到原文。这样eve收到的也是密文。这样保证了>机密性<
  • 但是eve如果想知道邮件的内容，他会想办法去还原。这个过程叫做>密码破译< 也叫>密码分析<。进行破译的这个人叫>破译者<，破译者不一定是坏人，也可以是坚定加密是否安全的工作人员。

2密码的简单概念

  • 解决复杂的问题的步骤叫算法。sender把邮件进行加密的步骤叫>加密算法<
    ，receiver收到邮件进行解密的步骤叫>解密算法<。加密和解密算法合起来统称密码算法。
  • 密码算法中需要>秘钥<。
  • 对称密码：加密和解密使用同一套秘钥。
  • 公有密码：也叫非对称密码，加密和解密使用不同的秘钥。
  • 混合密码：将对称密码和公有密码结合起来的加密方式。

3密码学家的工具箱 常用的密码技术

  • 密码技术不单单是基于密码的机密性。也可以检验消息的完整性，以及用于确认对方是否是本人的认证。
  • 单项散列函数：软件的发布者发布了软件和这个软件的散列值。下载软件的人自行计算下载软件的散列值。如果和发布者发布的一致，说明软件没有被篡改。这是为了保证文件的完整性。
  • 消息认证码：能保证完整性和提供认证的密码技术。
  • 数字签名：能确保完整性，提供认证并防止否认的密码技术。
  • 伪随机数生成器：一种模拟生成随机数的算法。比如web中进行ssl/tls通讯时，会产生一个仅用于当前会话的临时秘钥（会话秘钥）。
  • 对称密码
  • 公有密码

4密码和信息安全常识

  • 不要使用保密的密码算法。能公开密码算法让世界顶尖的黑客去破译的算法更安全。
  • 使用低强度的密码比不使用密码更危险。
  • 任何密码都有被破解的一天。
  • 密码只是信息安全的一部分。

—————–Git———-

文件如果之前已经被git跟踪了，这样修改是没有用的。每次修改完以后，用git status还是能看到提示它被修改了。

解决办法就是在git中删除这个文件的跟踪记录，用这个命令

git rm –cached 文件名路径和名称

这样就从git的跟踪记录中删除了这个文件的跟踪记录。配合之前在.gitignore加的那行配置，以后你修改 project.iws 这个文件，git就不会有提示了。

不想继续追踪某个文件

git update-index –assume-unchanged build.gradle

想再次继续跟踪某个文件

git update-index –no-assume-unchanged your_file_path

当你修改了一些文件，但还不能提交，于是需要保持当前的修改，先合并

git stash

打开保持的修改

git stash pop
