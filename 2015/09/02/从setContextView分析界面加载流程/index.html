从setContextView分析Android界面加载流程

发表于 2015-09-02   |    阅读次数

我们通过源码来了解view的加载流程。

首先我们加载布局是通过setContentView(R.layout.xxx);我们看setContentView这个方法的源码：

public void setContentView(int layoutResID) {
    getWindow().setContentView(layoutResID);---@1
    initActionBar();
}

@1：getWindow（）源码：

public Window getWindow() {
    return mWindow;
}

返回一个windows对象，这个windows是一个接口，下面是此类的注释：

/**
 * Abstract base class for a top-level window look and behavior policy.  An
 * instance of this class should be used as the top-level view added to the
 * window manager. It provides standard UI policies such as a background, title
 * area, default key processing, etc.
 *
 * <p>The only existing implementation of this abstract class is
 * android.policy.PhoneWindow, which you should instantiate when needing a
 * Window.  Eventually that class will be refactored and a factory method
 * added for creating Window instances without knowing about a particular
 * implementation.
 */

抽象的基类，提供最上层的展示窗口和表现方式。这个类的一个实例需要用作一个顶层的view被windowManager添加。它提供了标准的ui，像背景，标题，默认的key处理，等。

此类唯一的一个实现，就是android.policy.PhoneWindow。

那我们来看看phonewindows中的setContextView。

public void setContentView(int layoutResID) {
    if (mContentParent == null) {
        installDecor();
    } else {
        mContentParent.removeAllViews();
    }
    mLayoutInflater.inflate(layoutResID, mContentParent);--@2
    final Callback cb = getCallback();
    if (cb != null && !isDestroyed()) {
        cb.onContentChanged();
    }
}

if (mContentParent == null) ，mContentParent是什么？private ViewGroup mContentParent;是一个viewgroup。这个变量的注释：

// This is the view in which the window contents are placed. It is either
// mDecor itself, or a child of mDecor where the contents go.

这是一个view，被window包含的，它可以是一个mDecor，也可以是mDecor的子类。

那mDecor是谁：它是window的最顶层的view，包含于window。DecorView是一个framlayout。

// This is the top-level view of the window, containing the window decor.
private DecorView mDecor;

那就是说，一个activity都有一个window，也就是phonewindow，每一个phonewindow又包含一个decorview。

@2:将我们的布局添加到mContentParent中。
